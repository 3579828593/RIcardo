三、架构质量评估

3.1 架构优势

  • 安全措施到位 — CSRF 保护、pbkdf2_sha256 密码哈希（30 万次迭代）、参数化查询、安全响应头、限流、compare_digest 防时序攻击
  • ADR 决策记录完整 — 6 条架构决策含背景和取舍理由（Vue CDN vs SPA、Session 隔离、SW 策略等）
  • 多端部署支持 — PythonAnywhere / Render / Docker / 桌面 exe 四套部署方案
  • 测试覆盖面广 — 11 个测试模块 + 1 个 E2E，涵盖 API/认证/权限/迁移/CSV/前端
  • PWA 离线支持 — Service Worker StaleWhileRevalidate 策略，离线可答题

3.2 架构债务（按严重程度排序）

P0 — 严重（影响可维护性和安全性）
  • 双身份模型代码爆炸 — 匿名 session_id + 登录 user_id 并存，几乎所有数据层方法都有 if user_id: ... else: ... 双分支，代码量翻倍。get_stats 方法 60 行 4 分支极度臃肿。渗透到 record_answer、get_mistakes、get_favorites、toggle_favorite 等所有核心方法。
  • app.py 胖控制器（781 行） — 32 个路由 + 业务逻辑 + 判分 + 权限 + 会话管理混于一处，未使用 Flask Blueprint 拆分。
  • database.py 过载（884 行） — Schema + 4 步迁移 + 全部 CRUD 在一个类中，11 张表的所有操作集中一处。
P1 — 中等（应近期修复）
  • 前后端判分逻辑重复 — app.js:checkAnswerLocally 与 app.py:_check_answer 两套实现，存在不一致风险
  • 双重管理员机制 — require_admin（令牌鉴权）与 role='admin'（角色鉴权）两套独立体系并存
  • 迁移静默吞异常 — _pre_migrate/_post_migrate 的 except: pass 会掩盖迁移失败
  • 权限检查不一致 — 部分路由用 _check_bank_access 辅助函数，部分直接调 can_read_bank
  • 无测试覆盖率配置 — 无 pytest.ini / .coveragerc / pyproject.toml，无法量化覆盖率
P2 — 轻微（技术债务清单）
  • config.yaml 与 DEFAULT_CONFIG 完全重复，YAML 无增量价值
  • secret_key 随机回退无启动警告，生产环境忘记配置会导致 session 频繁失效
  • exam_sessions 表已定义但无任何 CRUD 方法（死代码）
  • CSP 允许 unsafe-eval（Vue CDN 模式所需，已知取舍）
  • 根目录存在 index.html、sw.js 等与 backend/ 下重复的冗余文件
  • import 位置违反 PEP8（from permissions import 在 app.py 第 591 行）

六、结论

回答用户的三个核心问题：

Q1: 我们有完整的开发文档和设计吗？

基本完整，但有缺口 项目拥有路线图、宪法、SDD 工作流、UGC 设计文档、6 条 ADR、代码审查报告、部署技能 — 文档体系在个人项目中已属优秀。但缺少独立 API 参考文档、系统架构图、测试覆盖率配置，且 progress.yaml 已过期。

Q2: 现在的推进和设计是否需要打磨？

需要针对性打磨 核心架构债务是双身份模型（session_id + user_id）导致的代码重复，渗透到数据层每个方法。app.py（781 行）和 database.py（884 行）两个大文件承载过多职责。建议按 P0→P1→P2 优先级逐步打磨，而非推倒重来。

Q3: 工作流是否需要优化和设计？

SDD 可用，SOLO 可选增强 SDD 工作流（SPEC→实现→REVIEW）已能支撑开发，但缺少自动化质量门禁（编码扫描、覆盖率检查、预部署验证）。SOLO 路由表和 quality-guard 模块在记忆中记录已建，但未落地到本项目。建议先补质量门禁，再评估是否引入 SOLO 路由。
